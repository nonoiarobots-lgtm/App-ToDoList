## 2. Risques restants, par priorité

### 🟠 Moyen

**R1 — Pas de sauvegarde de la base (plan Supabase Free).** Le plan gratuit n'inclut aucun backup automatique. Aujourd'hui la base est quasi vide ; dans 3 mois elle contiendra toute l'organisation personnelle de l'utilisateur. *Recommandation : dès que l'app est utilisée quotidiennement, mettre en place un export hebdomadaire (GitHub Action + `pg_dump` vers le repo privé ou un stockage), ou passer au plan Pro (25 $/mois, backups 7 jours).*

**R2 — Vulnérabilité modérée `postcss` via Next.js** (`npm audit` : XSS dans la sortie CSS, [GHSA-qx2v-qp2m-jg93](https://github.com/advisories/GHSA-qx2v-qp2m-jg93)). Le « fix » proposé par npm (rétrograder Next) est absurde ; le risque réel est faible (outil de build, pas d'exécution côté client de CSS non maîtrisé). *Recommandation : surveiller les releases Next 16.3.x et monter de version quand le correctif est intégré.*

**R3 — Protection « mots de passe compromis » désactivée** (advisor Auth). Supabase peut refuser les mots de passe présents dans HaveIBeenPwned. *Recommandation : activer dans Authentication → Passwords si disponible sur le plan Free ; sinon, risque accepté (app mono-utilisateur, mot de passe fort).*

### 🟡 Faible / accepté

- **Reset de mot de passe et PKCE** : le lien email doit être ouvert dans le **même navigateur** que la demande (le code verifier est en localStorage). Limitation standard Supabase, acceptable en solo — à documenter si ça surprend à l'usage.
- **Comparaison du `CRON_SECRET` non constant-time** dans `/api/keepalive` — théorique (timing attack sur un secret de 48 hex), négligeable ici.
- **Pas de rate-limiting applicatif** sur `/api/auth/*` — Supabase Auth applique ses propres limites ; suffisant en V1.
- **`confirm email` désactivé** : choix assumé (app solo). Une inscription tierce reste possible mais ne voit que ses propres données (RLS).
